Source relations

The "Source relations" block contain all the relations you want to use for your query. To add your own relations to the list, use the green-button relation element:

If you fill out both the name and the data and press the green add button, then the relation gets appended to the list of source relations. And it will look like this:

\
nameage
Anne19
Bob20
Claire21

If you double click on the table rows, then you enter editing mode.

Fill in the name of the relation

There are certain rules when setting your own source relations:

  • Every relation must have a name.
  • The relation name can only contain alfanumeric characters like A-Z, a-z or numbers, but it must start with a letter.
  • Relation names must be unique.

Fill in the rows of the relation

The data section can be either a CSV or a TSV (tab separated values) formatted text. The first line of text describes the attributes of the relations, while the rest of the lines represent the rows.

CSV example
name;age
Anne;19
Bob;20
Clair;21
TSV example
name	age
Anne	19
Bob	20
Clair	21

Although the TSV format looks more pleasing to the eye, typing tabs in textfields can turn out to be quite a hassle. TSV is useful mostly if you want to copy-paste the data from somewhere else, otherwise CSV is more practical.

Just like before, there are certain rules to filling out the data section:

Generally the data

  • The data should be valid CSV or TSV formatted
    • There is not supposed to be any delimiter at the end of the line. Although the parser ignores this error and processes the relation anyway.
    • There is not supposed to be a newline after the last line, as it would indicate a new, but empty row. The parser ignores the last row if it is empty.
  • The first row of data is required, and it defines the attributes.
  • The rest of the rows are optional, but they define the rows of the relation.

The attributes

  • Attribute names can only contain alfanumeric characters like A-Z, a-z or numbers, but they must start with a letter.
  • Attribute names must be unique within the relation.

The rows

  • The number of values defined in the row must match the number of attributes in the relation. The only exception to this is the trailing delimiter, because the parser ignores it as sloppy but correct.

Save or load relations

Setting up relations every time you want to use them, every time you revisit the page can be annoying. After adding your relation, you can use the button to save it to the browsers long term storage ( - not cookie!). This button will be available on your relations' headers.

To load a previously saved relation, you need to look for the "Source relations" panel. On its header, you will find a button. Clicking it will open a dialog, where you can load a relation, or delete it from the long term storage ( - still not a cookie!).

Writing and executing the query

When your relations are set, it is time to do some relational algebra. The middle section of the site is named just that. Find this block and write your own expression and press Go!

Let's say, you want to check the names in your relation, so you write the following query:PI ( name ; R1 )

If so, then after pressing 'Go', you will see a new relation appear in the 'Result relation' section, on the bottom third part of the site.

Result relation

Anne
Bob
Claire

Hooray. This is a relation too.

Double clicking on the table will switch to edit mode.

But if you want to use the results as a source for your next query, then simply press the green + button.

Finally, you will notice that a query tree appears in a block named just like that. It shows the graph of the query. You can drag the picture, and zoom in or out by clicking with left or right mouse button respectively.

In the following parts, you will learn more about how to write your own query.

Grammar introduction

Relational algebra uses operators that are coded by characters that are not available on normal keyboards. To make the writing of relational algebraic expressions possible, two different solutions were implemented. First, a grammar/syntax was created. If you are the type of person, who likes to learn programming/markup languages, you will find that there are easily typable symbols for every tricky operator, like @@ can stand for ⟗.

To provide another way, above the editor there is a toolbar with the operator symbols on it. Pressing any of these symbols inserts the symbol into the editor. That way it is possible to write queries that resemble the original algebraic expressions, like R ⟗ S.

So, there is the new grammar and the click-the-button-get-the-symbol approach. So in the end typing R @@ S or R ⟗ S will do exactly the same.

But in relational algebra there are operators that take parameters - usually in subscriptum (e.g.: 𝞼 name=Anne( R )). Currently it is not possible to type subscript texts in the query editor, so for these operators a new grammar to learn. Every operator that takes arguments has a parenthesis after its symbol, as if it was a function. Inside the function there are the parameter and the relation, relations. So, the query mentioned before would look like this in the editor: SIGMA ( name == 'Anne' ; R ) or σ ( name == 'Anne' ; R ).

That pretty much sums up everything you need to know. The intention was to make the grammar tokens resemble the original operators.

Grammar example
Operation Classical relational algebraic expression The same expression in the simulator
crossproduct R × S R × S
natural join R ⋈ S R @ S
full outer join R ⟗ S R @@ S
selection 𝞼 name=Anne( R ) SIGMA ( name == 'Anne' ; R )

As you can see, some of the operators are the same or quite similar in the grammar, while others had to be changed to make them keyboard friendly.

First and foremost: the result of every operation is a relation. That makes nesting possible, and it works exactly like the original algebra. It is also possible to use parenthesis to change the calculation order, because by default there is no precedence order between the operators.

The operators can be divided into groups based on their arity, or the number of extra parameters they take. Based on arity, the binary operators always look like this:

relation-name1 operator relation-name2

While unary operators always look like this:

operator ( parameter ; relation-name )

The only exception to these is the ϴ-join, which is a binary operation with parameters - and that makes it unique:

THETA_JOIN ( condition ; relation-name1, relation-name2 )

Detailed overview

Syntax of the syntax overview

A token is a nonterminal, i.e. something that is only a concept, not something you type. When you see relation, you can drill-down and check what it can be made of, and you will find that a relation can be a relation-name. Now the name of a relation is where you can stop, it is a terminal, i.e. something you can type. That's the difference between the colors of the tokens.

The black signs are not part of the grammar, i.e. you don't type them. They help to understand how the grammar builds up.

The (black) | sign shows that a given nonterminal can be a list of things, and the possible elements in the list are separated by this sign. The a | b means that either a or b can be used in that rule. The similarities to the regular expressions won't stop here.

The (black) parenthesis with an asterisk: ( ... )* shows that the content of the parenthesis can be omitted, used or even multiplied. The a ( , b )* means that a is syntactically correct, and so is a,b or a,b,b and so on.

The (black) bracket [ ... ] shows that the content can be omitted or used once, but not more often than that. It means, it is optional.

Now back to the grammar. A syntactically correct relational algebraic expression is a query.

query := relation

relation := relation-name | relation | ( relation ) | unary-operation | binary-operation

Binary operations

binary-operation := relation binary-operator relation | theta-join

theta-join := THETA_JOIN ( conditional ; relation , relation ) | ϴ ( conditional ; relation , relation )

binary-operator := + | - | * | × | ÷ | @ | $@ | @$ | !@ | &@ | @& | @@ | ⋃ | ⋂ | ⋈ | ⋉ | ⋊ | ▷ | ⟕ | ⟖ | ⟗

Unary operations

unary-operation := projection | selection | rename | distinct | sort | aggregate

projection := pi ( attribute-list ; relation ) | pi ( extended-attribute-list ; relation-name )

selection := sigma ( conditional ; relation-name )

rename := ro ( relation-name ; relation-name ) | ro ( rename-list ; relation-name )

distinct := delta ( relation-name )

sort := tau ( attribute-list ; relation-name )

aggregate := gamma ( aggregation-list ; relation-name )

pi := PI | π

sigma := SIGMA | σ

ro := RO | ρ

delta := DELTA | δ

tau := TAU | τ

gamma := GAMMA | γ

Now that we fixed the syntax of every operator, let's look into the syntax of their various arguments. The examples in the table of the 'Quick overview' section might prove to be useful in deciphering the rules.

General blocks

attribute-name := attribute-absolute-name | attribute-relative-name

attribute-absolute-name := relation-name . attribute-relative-name

attribute-relative-name := [A-Za-z][A-Za-z0-9]*

relation-name := [A-Za-z][A-Za-z0-9]*

calculation := ( calculation ) | literal ( arithmetic-operator literal )*

literal := attribute-name | string | number

arithmetic-operator := + | - | * | / | % | ^

string := single or double quoted arbitrary text, for example: 'text' or "text"

number := any kind of C-like number, for example: 5 or 1.3 or -4.5

Attribute list

attribute-list := attribute-name (, attribute-name )*

extended-attribute-list := extended-attribute-name (, extended-attribute-name )*

extended-attribute-name := calculation [-> attribute-relative-name]

Conditional

conditional := condition | ! conditinal | ( conditional ) | conditional ( logical-operator conditional )*

condition := calculation comparision-operator calculation

logical-operator := && | ||

comparision-operator := == | <= | >= | != | < | >

Rename list

rename-list := attribute-name / attribute-relative-name (, attribute-name / attribute-relative-name)*

Aggregation

aggregation-list := aggregation-item ( , aggregation-item )*

aggregation-item := extended-attribute-name | aggregation

aggregation := aggregation-function [-> attribute-relative-name]

aggregation-function := aggregation-function-min | aggregation-function-max | aggregation-function-sum | aggregation-function-avg

aggregation-function-min := MIN ( attribute-name )

aggregation-function-max := MAX ( attribute-name )

aggregation-function-sum := SUM ( attribute-name )

aggregation-function-avg := AVG ( attribute-name )
